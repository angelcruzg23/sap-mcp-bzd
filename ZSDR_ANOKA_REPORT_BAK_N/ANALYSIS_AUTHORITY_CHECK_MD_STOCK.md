# ZSDR_ANOKA_REPORT_BAK_N — Authority Check Fix for MD_STOCK_REQUIREMENTS_LIST_API

| Field | Value |
|---|---|
| Program | ZSDR_ANOKA_REPORT_BAK_N |
| Include | ZSDR_ANOKA_REPORT_BAK_N_F02 |
| Form | ZF_PRS_DATA |
| System | SAP ECC 6.0 EHP8 (BZD / Client 130) |
| Date | April 24, 2026 |

---

## 1. Problem Description

When executing transaction **ZSD_ORDER_TRACKING1** (program `ZSDR_ANOKA_REPORT_BAK_N`), users receive the following error message in the status bar during data processing:

> **"No authorization for W56LE601004 in plant 3096"**

This message appears immediately after pressing Execute (F8) on the selection screen, during the data retrieval and processing phase — not when interacting with the ALV output. The report displays an empty ALV grid with the error in the status bar, preventing the user from seeing any results.

The error corresponds to SAP standard message **E401(61)**: *"No authorization for &1 in plant &2"*.

---

## 2. Root Cause Analysis

### 2.1 Origin of the Message

The message is **not** generated by custom code in the program. It is raised internally by the standard SAP function module `MD_STOCK_REQUIREMENTS_LIST_API`, which performs its own authority check before returning MRP/stock data.

Internally, the FM validates the following authorization objects:

| Object | Fields | Purpose |
|---|---|---|
| `M_LFPL_ORG` | MDATL, PLSCN, WERKS | Long-term planning scenario access (only when `plscn` is supplied) |
| `M_MTDI_ORG` | MDAKT = 'A', WERKS, DISPO (DUMMY) | MRP list display authorization per plant |

Since the program does not pass a planning scenario (`plscn`), the FM always evaluates `M_MTDI_ORG`. When the user lacks this authorization for a given plant, the FM raises message E401(61) as a type E (error), which propagates to the status bar.

### 2.2 Code Location

The call is located in include `ZSDR_ANOKA_REPORT_BAK_N_F02`, inside `FORM zf_prs_data`, within the main `LOOP AT it_vbep`:

```abap
CALL FUNCTION 'MD_STOCK_REQUIREMENTS_LIST_API'
  EXPORTING
    matnr                    = wa_vbap-matnr
    werks                    = wa_vbap-werks
  TABLES
    mdpsx                    = it_mdpsx
  EXCEPTIONS
    material_plant_not_found = 1
    plant_not_found          = 2
    OTHERS                   = 3.
IF sy-subrc <> 0.
  " ← Empty block: error message leaks to status bar
ELSE.
  ...
ENDIF.
```

The empty `IF sy-subrc <> 0` block means the FM's internal error message is never handled and appears directly in the SAP GUI status bar.

### 2.3 Why This Started Happening

This behavior was introduced with **EHP8**. The FM `MD_STOCK_REQUIREMENTS_LIST_API` in EHP8 includes stricter authority checks that were not present in previous enhancement packages. Users who previously had no issues now encounter authorization failures for plants they are not explicitly authorized to view MRP data for.

### 2.4 Existing Authority Checks in the Program

The program already has three other authority check mechanisms, none of which address this specific issue:

| Form | Auth Object | Purpose | Behavior |
|---|---|---|---|
| `validate_authorization` | `M_MSEG_WWE` | STO supplying plant access | Shows popup with unauthorized plants, filters `r_reswk` range |
| `zauthority_check` | `M_MATE_VKO` | Sales organization access | Removes unauthorized records from `it_output`, stops if all removed |
| `zf_prs_data` (STO loop) | `M_MSEG_WWE` | STO supplying plant in PO processing | Skips record with `CONTINUE` |

None of these cover the `M_MTDI_ORG` object used by the MRP FM.

---

## 3. Proposed Solution

### 3.1 Approach

Add a pre-validation authority check for `M_MTDI_ORG` immediately before the call to `MD_STOCK_REQUIREMENTS_LIST_API`. This replicates the same check the FM performs internally, allowing the program to gracefully skip unauthorized material/plant combinations instead of letting the FM raise an unhandled error.

### 3.2 Code Change

**File:** `ZSDR_ANOKA_REPORT_BAK_N_F02`
**Form:** `ZF_PRS_DATA`
**Location:** Inside the `LOOP AT it_vbep`, before the `CALL FUNCTION 'MD_STOCK_REQUIREMENTS_LIST_API'`

**Current code:**

```abap
      ELSE.
        CALL FUNCTION 'MD_STOCK_REQUIREMENTS_LIST_API'
```

**Proposed code:**

```abap
      ELSE.
*       >>> BEGIN OF INSERTION - Authority pre-check before MD_STOCK_REQUIREMENTS_LIST_API
*       Replicate the same authority check the FM does internally
*       to avoid error message E401(61) for unauthorized material/plant combinations
        AUTHORITY-CHECK OBJECT 'M_MTDI_ORG'
          ID 'MDAKT' FIELD 'A'
          ID 'WERKS' FIELD wa_vbap-werks
          ID 'DISPO' DUMMY.

        IF sy-subrc NE 0.
*         User has no MRP authorization for this plant - skip to next record
          CLEAR ls_mdps.
          CONTINUE.
        ENDIF.
*       <<< END OF INSERTION - Authority pre-check

        CALL FUNCTION 'MD_STOCK_REQUIREMENTS_LIST_API'
```

### 3.3 Rationale

| Decision | Reason |
|---|---|
| Only check `M_MTDI_ORG`, not `M_LFPL_ORG` | The program never passes a planning scenario (`plscn`) to the FM, so the long-term planning check is never triggered |
| `MDAKT = 'A'` | Activity 'A' corresponds to "Display MRP list", which is what the FM uses |
| `DISPO DUMMY` | The FM does not restrict by MRP controller — using DUMMY matches this behavior |
| `CONTINUE` on failure | Skips the current record silently, consistent with how the STO authority check already works in the same form |
| No error message on skip | The record simply won't show unrestricted stock data; the user sees the rest of the report normally |

---

## 4. Behavior After the Change

### 4.1 Before (Current Behavior)

1. User executes the report with selection criteria
2. During `zf_prs_data`, the FM `MD_STOCK_REQUIREMENTS_LIST_API` is called for each material/plant
3. If the user lacks `M_MTDI_ORG` authorization for any plant, the FM raises message E401(61)
4. The error message appears in the status bar: *"No authorization for [material] in plant [plant]"*
5. The ALV may display incomplete or empty results
6. The last error message remains visible, potentially confusing the user

### 4.2 After (Expected Behavior)

1. User executes the report with selection criteria
2. During `zf_prs_data`, before calling the FM, the program checks `M_MTDI_ORG` for each material's plant
3. If the user lacks authorization for a specific plant:
   - The record is skipped silently (`CONTINUE`)
   - No error message is displayed
   - Processing continues with the next record
4. The ALV displays all records the user is authorized to see
5. Records for unauthorized plants are excluded — the "Unrestricted Stock Qty" column will simply not appear for those entries
6. No disruptive error messages in the status bar

### 4.3 Impact Assessment

| Aspect | Impact |
|---|---|
| Functional | Records for unauthorized plants are excluded from stock quantity lookup only; all other fields (order, delivery, shipment data) remain unaffected |
| Performance | Negligible — `AUTHORITY-CHECK` is an in-memory operation, much faster than the FM call it prevents |
| Security | No change — the same authorization object is validated; the check is simply moved earlier in the flow |
| User experience | Significant improvement — no more confusing error messages blocking the report |
| Regression risk | Low — the change only adds a guard clause before an existing FM call |
